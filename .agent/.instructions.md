# emergent-sim Development Instructions

This file guides AI agents working on the emergent-sim project.

## Project Overview

**emergent-sim** is a 3D ECS simulation engine built with Rust and Bevy 0.15.

- **Goal**: Technical portfolio piece demonstrating system design, not a finished product
- **Architecture**: 5 strict layers (engine → simulation → AI → observability → scenarios)
- **Key Challenge**: Maintain absolute separation of concerns between layers
- **Test Focus**: Unit + integration + determinism tests (80%+ coverage required)

## Architecture Constraints (MUST FOLLOW)

### Layer Hierarchy
```
engine/      — Rendering, input, camera, ECS loop (NO simulation imports)
simulation/  — World logic, agents, resources (NO engine imports)
ai/          — Decision making (read-only access to simulation)
observability/ — Debug tools (read-only access to all)
scenarios/   — Configuration and presets (read-only access to simulation)
```

### Forbidden Patterns
1. **NO upward dependencies** — simulation cannot import engine
2. **NO direct system calls** — use events only
3. **NO logic in components** — components = data only
4. **NO hardcoded simulation values** — use resources + config
5. **NO simulation mutations from observability** — read-only
6. **NO SimRng outside simulation/ai** — seeded RNG in sim layer only

### Good Patterns
- Event-driven communication between layers
- Configuration in RON assets (hot-reloadable)
- Pure utility functions for scoring
- Resource-based global state
- Comprehensive testing at all levels

## When Working on This Project

### Before Writing Code
1. Identify which layer the code belongs in
2. Check ARCHITECTURE.md for system ordering
3. Verify no rule violations
4. Plan testing strategy

### Code Quality Checks (REQUIRED)
```bash
cargo fmt --all
cargo clippy --all-targets --all-features -- -D warnings
cargo test --all-features
cargo test determinism --verbose
```

### Common Mistakes to Avoid
- ❌ Importing engine in simulation code
- ❌ Calling systems directly instead of via events
- ❌ Adding logic methods to components
- ❌ Hardcoding needs decay rates in systems
- ❌ Mutating simulation state from observability layer
- ❌ Using system clock time instead of SimRng

### Documentation
- All public items need doc comments
- All systems need one-line doc comment
- Update ARCHITECTURE.md if structure changes
- Update ECS_DESIGN.md if components/systems/events change

## Testing Strategy

### Unit Tests (Pure Functions)
- Utility scoring functions
- Curve evaluations
- Need calculations
- Grid queries

### Integration Tests
- Agent eating → hunger relief
- Resource depletion → regeneration
- Agent death mechanics
- Scenario loading

### Determinism Tests (Critical)
- Same seed → identical output
- Must account for RNG consumption
- Any non-determinism is a **blocking bug**

### Coverage Target
- Minimum 80% code coverage
- All public APIs tested
- All error paths tested

## Common Questions

**Q: How do I add a new component?**
A: Add to appropriate module, register in plugin, update ECS_DESIGN.md, write tests.

**Q: How do I add a system?**
A: Write function with `_system` suffix, register in correct SystemSet, update ECS_DESIGN.md, write tests.

**Q: Can I add a dependency?**
A: Only if justified. Prefer using Bevy's ecosystem (bevy-inspector-egui, bevy-gizmos, etc.).

**Q: What if I need simulation state in the engine?**
A: Use events to signal from simulation to engine, or read resources (immutable).

**Q: When should I use features?**
A: `observability` and `debug_overlays` are already features. All debug tools are gated.
